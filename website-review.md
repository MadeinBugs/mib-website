## 1. Project Overview

**Made in Bugs** is an indie game development studio founded in 2024. This is their main website (`madeinbugs.com.br`), serving as:

- A **public portfolio** showcasing the studio and its game projects.
- A **newsletter/mailing list** system for the Asumi game pre-launch campaign.
- An **internal mascot customization tool** where team members customize the studio mascot (Sisyphus the dung beetle) for a yearly team poster.
- A **picture contest gallery** used during indie game festivals to let players view and favorite photos taken in-game.

The studio mascot Sisyphus is a dung beetle, symbolizing perseverance through "bugs" — a play on the studio's name. The brand identity is minimalistic and hand-drawn, inspired by "Baba Is You."

**Team size at time of writing:** 7 members (including Sisyphus the mascot as "Game Tester").

## 4. Routing Architecture

### App Router Structure

The site uses Next.js **App Router** with a locale-based dynamic segment:

```
/                           → Redirects to /pt-BR
/[locale]/                  → Homepage (locale = 'en' | 'pt-BR')
/[locale]/about             → About / Team
/[locale]/contact           → Contact
/[locale]/portfolio         → Project grid
/[locale]/projects/[id]     → Individual project page
/[locale]/jobs/[id]         → Job detail page
/[locale]/bugsletter        → Newsletter signup (Asumi-focused)
/[locale]/newsletter/confirmed       → Subscription confirmation landing
/[locale]/newsletter/unsubscribe     → Unsubscribe page
/[locale]/picture-contest            → Contest entry (code input)
/[locale]/picture-contest/login      → Admin login
/[locale]/picture-contest/gallery    → Admin gallery (auth-required)
/[locale]/picture-contest/[uniqueId] → Player photo gallery
/[locale]/picture-contest/logout     → Sign out
/mascot                     → Mascot editor (auth-required)
/mascot/login               → Mascot login
/mascot/register            → Mascot registration (invite code)
/mascot/gallery             → Artist gallery (role: artist only)
/mascot/logout              → Sign out
/mascot/api/validate-invite → POST: validate invite code
/mascot/api/consume-invite  → POST: consume invite + create profile
/privacy                    → Privacy policy
/terms                      → Terms of service
/api/newsletter/subscribe   → POST: newsletter signup
/api/newsletter/confirm     → GET: email confirmation
/api/newsletter/unsubscribe → POST: unsubscribe
/api/contest/validate-code  → GET: validate session code
/api/contest/choose-favorite → POST: mark photo as favorite
/api/dev/preview-email      → GET: dev-only email preview
```

### Key Routing Decisions

- **Mascot routes are NOT under `[locale]`:** The mascot editor is an internal tool; it handles its own i18n via `MascotLocaleContext` with `localStorage` persistence instead of URL-based locale.
- **Picture contest IS under `[locale]`:** It's player-facing during festivals, so URL-based locale matters.
- **Privacy/Terms are NOT under `[locale]`:** Simple English text, no i18n.
- **`mib-social-manager/` exists as an empty directory** — planned but unimplemented feature.

## 5. Internationalization (i18n)

### Supported Locales

- **pt-BR** (Brazilian Portuguese) — default
- **en** (English)

### Implementation

There is **no i18n library** (no `next-intl`, `react-i18next`, etc.). It's a custom implementation:

- **Main site translations:** `messages/en.json` and `messages/pt-BR.json`, loaded by `src/lib/i18n.ts` with in-memory caching.
- **Mascot translations:** `src/lib/mascotI18n.ts` — hardcoded bilingual object with `MascotLocaleContext` (localStorage-persisted).
- **Picture contest translations:** `src/lib/pictureContestI18n.ts` — same pattern as mascot.
- **Job listings:** Bilingual title/description objects inline in `src/lib/jobs.ts`.
- **Projects data:** Bilingual subtitle/description/features/challenges inline in `src/lib/projects.ts`.
- **Email content:** Separate markdown files per locale (`welcome.en.md`, `welcome.pt-BR.md`).

### Locale Switching

- Main site: `LanguageSwitcher` component with flag emoji buttons (🇧🇷 / 🇺🇸), preserves current path.
- Mascot: `MascotLanguageSwitcher` component, stores preference in `localStorage`.
- Picture contest: `PictureContestLayoutClient` includes a language switcher.

### URL Normalization

Middleware handles case-insensitive locale matching:
- `/pt-br/*` → 308 redirect to `/pt-BR/*`

### Translation Structure (messages/*.json)

```
navigation: { about, portfolio, contact }
homepage: { title, subtitle, description }
pages: { about: {...}, portfolio: {...}, contact: {...} }
ui: { language, loading, error }
common: { languageSwitcher }
newsletter: { title, placeholder, submit, success, error, alreadySubscribed, rateLimit }
bugsletter: { title, subtitle, cta, signupTitle, signupSubtitle, ... }
```

## 10. Newsletter / Bugsletter System

### Overview

Double opt-in newsletter system built around **Brevo** (formerly Sendinblue) for the Asumi pre-launch mailing list campaign.

### Brevo Configuration

- **API URL:** `https://api.brevo.com/v3`
- **3 mailing lists:**
  - List 5: Asumi (automatic for all subscribers)
  - List 6: Studio (opt-in, future use)
  - List 7: Devlog (opt-in, future use)
- **Custom contact attributes:** `CONFIRMED` (boolean), `CONFIRMATION_TOKEN` (UUID), `CONFIRMATION_CREATED_AT` (ISO date), `LOCALE`.

### Subscription Flow

1. **User submits email** on `/[locale]/bugsletter` page via `NewsletterSignup` component.
2. **POST `/api/newsletter/subscribe`:**
   - Rate limit: 3 requests per IP per 60 seconds.
   - Honeypot field check (bot protection).
   - Email validation.
   - Creates/updates Brevo contact with confirmation token (UUID).
   - Sends confirmation email (7-day TTL).
   - Posts Discord webhook notification.
   - **If already confirmed:** Returns success silently (no information leak).
   - **If unsubscribed:** Respects decision, returns success silently.
3. **User clicks confirmation link** → **GET `/api/newsletter/confirm`:**
   - Validates token against Brevo contact.
   - Checks 7-day expiration.
   - Sets `CONFIRMED: true`.
   - Sends welcome email.
   - Redirects to `/[locale]/newsletter/confirmed?status=success|expired|invalid|error`.
4. **Confirmation page** shows status-appropriate message with themed card.

### Unsubscribe Flow

1. Every email includes an unsubscribe link with **HMAC-SHA256 signature** (using `UNSUBSCRIBE_SECRET`).
2. `/[locale]/newsletter/unsubscribe` page extracts `email` + `sig` from query params.
3. Client component shows confirmation UI → **POST `/api/newsletter/unsubscribe`:**
   - Rate limit: 5 requests per IP per 60 seconds.
   - Validates HMAC signature with timing-safe comparison.
   - Removes contact from all newsletter lists in Brevo (does NOT delete the contact record).

### Frontend Component: `NewsletterSignup`

- Email input with regex validation (`.+@.+\..{2,}`).
- Honeypot hidden field.
- States: idle, loading, success, error.
- Rate limit handling (429 response).
- **Umami analytics event** on successful signup.
- Customizable placeholder, button text, accent color (reusable across pages).

### Documentation

Full plan in `MailingListPlan.md`.

## 12. Mascot Customization Editor

### Overview

A **Windows XP Paint-themed** browser-based image editor where team members customize Sisyphus (the studio mascot) for a yearly team poster. Protected by invite-code registration + Supabase auth.

### Technical Implementation

**Canvas engine:** Konva.js (`react-konva`) with a 1024×1024 canvas.

**7-layer architecture:**

| Layer | Content | Editable |
|---|---|---|
| 0 | Tinted body + back PNGs | No (color/opacity adjustable) |
| 1 | Tinted eyes PNG | No (color/opacity adjustable) |
| 2 | SVG patterns clipped to region masks | No (pattern selection + settings) |
| 3 | Outlines & details | No (locked) |
| 4–6 | User-drawable raster layers | Yes (brush, eraser, stamp, text) |

**Region customization:**
- 3 regions: body, back, eyes.
- Each region: color picker + hex input, opacity slider.
- Body and back additionally support pattern overlays (squiggly, stripes, dots, stars) with pattern color, opacity, and rotation controls.

**Drawing tools:**
- Brush (variable size 1–50px, color, opacity).
- Eraser (variable size).
- Color picker (eyedropper reads pixel data from canvas).
- Stamp (shapes placed on click).
- Text tool ("A" button — stamps a large "A" character).
- Undo/Redo (Ctrl+Z / Ctrl+Y, max 50 history states).

**Drawable layer mask modes:**
- Unmasked (default).
- Mask In (paint only inside silhouette).
- Mask Out (paint only outside silhouette).

**Other features:**
- 🎲 Randomize button: randomizes all colors/patterns from 20 approved colors.
- Auto-save with status indicator → saves to Supabase.
- Preview PNG upload to Supabase storage.
- Export at 1x/2x/4x resolution (1024/2048/4096px).
- Commentary field (500 char limit).
- Year-scoped: each customization is unique per user per year.

### Windows XP Theme (`editor/xpStyles.ts`)

- Main color: `#D4D0C8`.
- Tahoma font.
- 3D beveled borders (raised/sunken helper functions).
- XP-style title bar, menu bar, toolbar, status bar, color palette components.

### Auth & Registration Flow

1. Admin generates invite codes in Supabase (`invite_codes` table).
2. Team member navigates to `/mascot/register`.
3. Enters: invite code, display name, email, password.
4. **POST `/mascot/api/validate-invite`** — server-side validation (uses service role key to bypass RLS).
5. Supabase auth account created.
6. **POST `/mascot/api/consume-invite`** — atomically increments usage + creates profile.
7. Redirected to `/mascot` editor.

### Artist Gallery (`/mascot/gallery`)

- Protected: requires `role: "artist"`.
- Displays all mascot customizations for the current year.
- Shows preview images (signed URLs from Supabase storage), creator name, last updated, commentary.
- JSON download option for customization data.

### Documentation

- Architecture plan: `MascottCustomization.md`
- Editor spec: `MascotCustomizationEditor.md`

## 13. Picture Contest System

### Overview

Photo contest system for **indie game festivals**. Players take photos during gameplay, view them via a unique session code, and pick favorites. Admins manage the gallery with a separate login.

### Player Flow

1. Player receives a 5-letter code during gameplay (format: consonant-vowel-consonant-vowel-consonant, e.g., `BAKOM`).
2. Player goes to `/[locale]/picture-contest`, enters code in `CodeEntryForm`.
3. **GET `/api/contest/validate-code`** — validates format and checks Supabase.
4. Redirected to `/[locale]/picture-contest/[uniqueId]` — `PlayerGallery` shows their session photos.
5. Player can mark up to **2 favorites** per session.
6. **POST `/api/contest/choose-favorite`** — validates, inserts into `contest_favorites`, posts image to Discord webhook.

### Admin Flow

1. Admin logs in at `/[locale]/picture-contest/login` (Supabase auth).
2. `/[locale]/picture-contest/gallery` — `AdminGallery` shows all sessions with thumbnails, photo counts, favorite counts.
3. Session cards link to individual session detail views.

### Feature Gate

- **`PICTURE_CONTEST_LIVE` env var:** When `false`, public contest routes are blocked (middleware redirects to login). Only authenticated admins can access. When `true`, all routes are public.

### UI Theme

- **Polaroid cards** (`PolaroidCard`): Random rotation based on ID hash, straightens on hover.
- **Cream background** (`#f7fff0`).
- **Heart sticker overlay** on favorited photos.
- **Confirmation modal** with Framer Motion animations.
- **Pangolin + Amatic SC fonts** for playful aesthetic.

### Documentation

Implementation plan: `Gallery_Photo_Contest.md`.

## 14. Supabase Architecture

### Two Separate Supabase Projects

| Project | Purpose | Env Var Prefix |
|---|---|---|
| Main | Mascot auth, profiles, customizations, invites | `NEXT_PUBLIC_SUPABASE_*`, `SUPABASE_SERVICE_ROLE_KEY` |
| Picture Contest | Contest sessions, photos, favorites | `NEXT_PUBLIC_PICTURE_CONTEST_SUPABASE_*`, `PICTURE_CONTEST_SUPABASE_SERVICE_ROLE_KEY` |

### Client Types (5 total)

| File | Side | Project | Auth |
|---|---|---|---|
| `lib/supabase/client.ts` | Browser | Main | Anon key |
| `lib/supabase/server.ts` | Server | Main | Anon key + cookies |
| `lib/supabase/picture-contest-client.ts` | Browser | Contest | Anon key |
| `lib/supabase/picture-contest-server.ts` | Server | Contest | Anon key + cookies |
| `lib/supabase/picture-contest-service.ts` | Server only | Contest | **Service role key** |

### Database Schema (Main — Mascot)

- **`invite_codes`**: id, code (unique), max_uses, current_uses, created_at, expires_at.
- **`profiles`**: id (UUID, FK → auth.users), display_name, role, invite_code_used, created_at.
- **`mascot_customizations`**: id, user_id (FK, unique per year), year, customization_data (JSONB), updated_at.

### Database Schema (Picture Contest)

- **`contest_sessions`**: unique_id, created_at, game_version, machine_id, booth.
- **`contest_pictures`**: id, unique_id (FK), filename, storage_path, picture_index, taken_at, uploaded_at, metadata, image_url.
- **`contest_favorites`**: id, session_unique_id, picture_id, discord_message_id, created_at.

### Storage

- **Mascot:** Preview PNGs uploaded to storage bucket with signed URLs.
- **Contest:** Private bucket for player photos, signed URLs (7-day expiration) generated server-side.

### Security

- **Row Level Security (RLS)** on all tables: users can only read/write their own data.
- **Service role key** used only server-side for admin operations (invite validation/consumption, bypassing RLS).
- Atomic operations with optimistic concurrency for invite consumption.


**Functions:**

1. **Locale normalization:** `/pt-br/*` → `/pt-BR/*` via 308 redirect.
2. **Supabase session refresh:** Refreshes auth tokens via cookies on every request.
3. **Mascot auth gate:** Unauthenticated users → `/mascot/login`. Public routes: `/mascot/login`, `/mascot/register`.
4. **Picture contest gate:**
   - If `PICTURE_CONTEST_LIVE=true`: All routes accessible.
   - If `false`: Only authenticated admins can access. Unauthenticated → `/[locale]/picture-contest/login`.
5. **Two separate Supabase clients:** Middleware creates different clients for mascot vs. contest auth checks.

### Auth Approach

- **Supabase Auth** with email + password.
- **Cookie-based sessions** managed by `@supabase/ssr`.
- **No OAuth/social login.**
- **No email verification** for mascot (invite codes provide the gate).
- **Mascot roles:** `profiles.role` field; `"artist"` role required for gallery access.

## 20. Security Decisions

### Implemented

- **HMAC-SHA256 signed unsubscribe URLs** with timing-safe comparison to prevent timing attacks.
- **Honeypot field** on newsletter form for bot protection.
- **Rate limiting** on newsletter subscribe (3/min) and unsubscribe (5/min) endpoints.
- **Supabase RLS** on all database tables.
- **Service role key** isolated to server-only code paths.
- **LGPD/GDPR compliance:** Double opt-in, unsubscribe links in every email, no email existence leaking.
- **Invite code system** for mascot registration (prevents unauthorized access).
- **Middleware auth gates** for protected routes.
- **No information leaking:** Subscribe endpoint returns success even for already-confirmed or unsubscribed emails.
- **Token expiration:** 7-day TTL on confirmation tokens.
- **Feature gate:** `PICTURE_CONTEST_LIVE` controls public access to contest.

### Concerns

- **Rate limiter is in-memory** (`src/lib/rate-limit.ts`): Does NOT persist across Vercel serverless invocations. Each cold start gets a fresh rate limit store. The code itself contains a note recommending Upstash Redis.
- **Contest API routes lack rate limiting:** `/api/contest/choose-favorite` and `/api/contest/validate-code` have no rate limiting.
- **Contest favorite endpoint has no auth check:** Any client can POST favorites if they know a session ID and picture ID.
- **No CSRF protection** on API routes.

## 21. Known Technical Debt

### High Priority

1. **In-memory rate limiter is ineffective on Vercel serverless.** Each function invocation may get a new instance with empty rate limit store. Recommended fix: Upstash Redis or Vercel KV.

2. **No `.env.example` file.** New developers must guess required environment variables from code. There are at least 10+ env vars needed across Supabase (×2 projects), Brevo, unsubscribe secret, site URL, Umami, picture contest live flag, and Discord webhook URLs.

3. **No testing infrastructure.** No unit tests, integration tests, or E2E tests exist. No test runner is configured.

4. **PWA manifests are placeholder/generic.** `src/app/manifest.json` has `name: "MyWebSite"`, `short_name: "MySite"`. `public/site.webmanifest` has empty name/short_name. Both should be updated to "Made in Bugs" branding.

### Medium Priority

5. **Repo memory is stale.** The `.memories/repo/mib-website-architecture.md` says "No API routes", "No database", "Deployment: GitHub Pages", "No Authentication" — all of which are now wrong.

6. **Images are unoptimized.** `next.config.js` has `images: { unoptimized: true }`, serving raw images without Vercel's image optimization pipeline. This affects load times especially for the 23 Asumi assets.

7. **Inconsistent i18n approach.** There are 4 different translation mechanisms:
   - JSON files + `getTranslations()` for main site.
   - Hardcoded bilingual objects in `mascotI18n.ts` and `pictureContestI18n.ts`.
   - Inline bilingual objects in data files (`jobs.ts`, `projects.ts`).
   - Separate Markdown files per locale for emails.
   
   No single i18n library unifies these.

8. **`mib-social-manager/` is an empty directory** — leftover from a planned feature. Should be removed or implemented.

9. **GitHub Actions workflow is backed up (`.yml.bak`)** — dead file from the pre-Vercel era. Should be cleaned up.

10. **`fix-contact.ps1` PowerShell script** exists as a one-time patch script. Should be removed if already applied.

11. **`sampa-games.png~` backup file** in `public/assets/about-us/` — editor artifact that should be removed.

12. **Version in `version.json` is from January 2025** (`26-01-2025-01`). The auto-versioning script may not be running in the Vercel pipeline.

### Low Priority

13. **README.md is outdated.** Still references GitHub Pages deployment and doesn't document Supabase, Brevo, mascot, or contest features.

14. **Some fonts are placeholders.** `tailwind.config.js` uses `Comic Sans MS` as placeholder for `crayon` and `playful` font families.

15. **No error boundary components.** React error boundaries are not implemented for graceful failure handling.

16. **No loading states for route transitions.** No `loading.tsx` files exist in the app directory.

17. **`getImagePath` and `getMetadataAssetPath` utilities are nearly identical** — both just normalize leading slashes. Could be consolidated.

18. **Tailwind safelist is verbose.** The extensive safelist for gradient classes suggests the dynamic class generation in `styles.ts` could be refactored to use Tailwind's JIT-friendly patterns.

19. **`postcss.config.js` and `tailwind.config.js` use CommonJS** (`module.exports`) while the rest of the project is TypeScript/ESM.

20. **`next.config.js` is also CommonJS** — could be migrated to `next.config.mjs` or `next.config.ts`.
